# 0. 이 프로젝트의 학습 목표

이 프로젝트 하나로 아래 4가지를 연결해서 이해하는 것이 목표입니다.

1. **Spring Boot IoC / DI / Bean**
2. **React의 상태 관리와 Hook**
3. **TanStack Query를 이용한 서버 상태 관리**
4. **React 렌더링 최적화의 실제 감각**

즉,

- 백엔드에서 객체를 누가 만들고 연결하는가?
- 프론트엔드에서 state가 바뀌면 화면은 어떻게 다시 그려지는가?
- 서버 데이터와 로컬 UI 데이터는 왜 다르게 다뤄야 하는가?
- 함수와 객체의 참조가 왜 렌더링 성능에 영향을 주는가?

이 질문에 대한 감각을 얻는 것이 핵심입니다.


> ### 📌 참고: Bean이란?
>
> 클래스에 new를 사용해서 생성하는 객체와는 달리, Java Spring 프레임워크상에서 IoC 컨테이너를 통해 생성하는 객체를 의미함 
>
> ---
>
> **1. 📤 출력 관점: Bean (빈)**
>
> - **정의**: IoC 컨테이너가 소스 코드를 읽어와 **실제로 힙(Heap) 메모리에 생성하고 관리하는 '객체(인스턴스)'** 만 Bean이라고 부릅니다.
> - **비유**: 공장에서 설계도를 보고 찍어낸 **실제 완성된 스마트폰 제품**이 바로 Bean입니다.
>
> ---
>
> **2. 📥 입력 관점: Bean Definition (빈 정의) 또는 Component (컴포넌트)**
>
> - **정의**: `@Component`, `@Service` 등을 붙여서 IoC 컨테이너에 등록하고자 대기시키는 **'클래스(설계도)'** 는 빈이 아니라 **'컴포넌트'** 또는 스프링 내부 용어로 **'Bean Definition(빈 정의 설계도)'** 라고 부릅니다.
> - **비유**: 스마트폰을 만들기 위해 공장에 제출한 **종이 설계도**입니다. 이 설계도 자체를 스마트폰(Bean)이라고 부르지는 않는 것과 같습니다.
>
> ---
>
> **한 줄 정리**
> - **클래스(설계도)** → Bean Definition / Component
> - **컨테이너가 만든 객체(인스턴스)** → Bean

> ### 🎯 왜 "IoC 컨테이너를 통한 Bean의 생성과 수명주기 관리"가 Spring의 핵심인가?
>
> 스프링 공식 문서에서도 스프링의 **핵심 심장(Core)** 을 정의할 때 정확히 **IoC 컨테이너**, 그리고 그 안에서 살아 숨 쉬는 **Bean의 생성과 생명주기 관리**를 꼽습니다.
>
> 왜 이것이 스프링의 전부라고 해도 과언이 아닌지, 그 핵심 이유를 **3가지**로 압축해 봅니다.
>
> ---
>
> #### 1️⃣ 개발자를 "반복 노동"에서 해방시켰기 때문
>
> 스프링이 없던 시절, 개발자들은 시간의 절반을 다음 인프라 코드에 썼습니다.
> - 수백, 수천 개의 클래스를 직접 `new`로 생성
> - 한 객체를 다른 객체에 일일이 주입
> - 다 쓴 객체를 안전하게 소멸
>
> 스프링의 선언:
> > **"너희는 핵심 비즈니스 로직(결제, 회원가입 등)만 짜라.  
> > 객체를 만들고 엮고 부수는 지루한 일은 내(IoC 컨테이너)가 전담하겠다."**
>
> 이 한 마디가 **자바 진영의 대혁신**이 되었습니다.
>
> ---
>
> #### 2️⃣ "생명주기 관리"가 왜 핵심 중의 핵심인가
>
> 단순히 객체를 `new`로 만드는 것은 누구나 할 수 있습니다.  
> 진짜 기술은 **객체가 태어나고(초기화) 죽을 때(소멸) 안전하게 제어하는 것**에 있습니다.
>
> | 단계 | 컨테이너가 보장하는 것 |
> |---|---|
> | **태어날 때** (`@PostConstruct`) | 객체 생성 직후 DB 연결을 맺거나, 외부 시스템과 미리 악수(Handshake)를 나누는 초기화 작업 보장 |
> | **일할 때** | 수많은 스레드가 들어와도 **싱글톤**으로 안전하게 **무상태(Stateless) 로직** 수행 |
> | **죽을 때** (`@PreDestroy`) | 서버 종료 시 Bean들을 차례로 정리하며 DB 연결을 안전하게 끊어(Connection Close) **데이터 유실·메모리 누수 방지** |
>
> ---
>
> #### 3️⃣ 스프링의 모든 부가 기능은 '여기'서 출발한다
>
> 우리가 감탄하는 다음 기능들:
> - 트랜잭션 관리 (`@Transactional`)
> - 보안 (Spring Security)
> - 로깅 / 성능 모니터링
>
> 이 모두는 **IoC 컨테이너가 Bean을 직접 쥐고 관리하기 때문에** 가능합니다.
>
> 컨테이너가 Bean의 생성과 수명주기를 통제하고 있기 때문에, **중간에 가짜 객체(Proxy)를 끼워 넣거나 부가 기능을 가로채기(AOP) 편한 구조**가 완성되는 것입니다.
>
> ---
>
> #### 📌 한 줄 요약
>
> > **"IoC 컨테이너 = 무대", "Bean = 배우"**
>
> 무대 감독(IoC 컨테이너)이 배우(Bean)를:
> - **언제 무대에 올리고** (생성/초기화)
> - **어떻게 연기시키고** (DI/의존성 주입)
> - **언제 퇴장시킬지** (소멸)
>
> 총괄하는 이 메커니즘이 바로 **스프링의 정체성이자 알파와 오메가**입니다.

        /api/tasks
           ↓

## 1-1. 데이터베이스: 일단 H2를 쓰는 이유

이 프로젝트의 DB는 **MySQL이 아니라 H2 인메모리 DB**입니다.

실무에서는 MySQL이나 PostgreSQL을 많이 쓰지만, 이 예제는 **Spring IoC / JPA / CRUD 흐름**에 집중하기 위해 일단 H2로 시작합니다.

- 별도 DB 서버 설치·실행 없이 Maven만으로 바로 실행 가능
- 애플리케이션이 뜰 때 메모리 위에 DB가 만들어지고, 종료하면 데이터도 함께 사라짐 (학습·실험에 적합)
- JPA·Repository·Entity 연습에는 H2와 MySQL의 사용 방식이 거의 같음

설정은 `backend/src/main/resources/application.yml`에 있습니다.

```yaml
spring:
  datasource:
    url: jdbc:h2:mem:studydb
    driver-class-name: org.h2.Driver
    username: sa
```

- **H2 콘솔**: `http://localhost:8080/h2-console`
- **JDBC URL** (콘솔 접속 시): `jdbc:h2:mem:studydb`
- **사용자명**: `sa` / **비밀번호**: (비워 둠)

`ddl-auto: create`로 되어 있어 **백엔드를 재시작할 때마다 테이블이 새로 만들어집니다.** 이전에 넣었던 할 일 데이터가 사라지는 것이 정상입니다.

나중에 실제 DB로 바꿀 때는 이 문서 맨 아래 **「17. 다음 단계 추천」**의 PostgreSQL 연동 항목을 참고하면 됩니다.

# 2. 왜 이 프로젝트가 좋은 뼈대인가

초보자는 종종 이렇게 배웁니다.

- React 따로
- Spring 따로
- fetch 따로
- Hook 따로
- 최적화 따로

이렇게 배우면 개념은 조각나는데, **실제 서비스 한 덩어리**로 연결되는 감각이 잘 안 생깁니다.

이 프로젝트는 그 조각들을 한 번에 연결합니다.

- Spring에서는 **IoC 컨테이너가 객체를 생성/주입**
- React에서는 **Hook이 렌더링 생명주기 안에서 state/side effect를 관리**
- TanStack Query는 **서버 상태를 캐시하고 동기화**
- React.memo, useMemo, useCallback은 **참조 안정성과 불필요한 재계산/재렌더를 줄임**

즉,  
이 프로젝트는 단순 CRUD 이상으로  
**프레임워크가 개발자의 제어를 어떻게 가져가는지**를 보여줍니다.

# 3. IoC, DI, Container를 가장 쉽게 이해하기

## 3-1. IoC란 무엇인가

IoC(Inversion of Control)는 직역하면 **제어의 역전**입니다.

보통 객체지향 초급 코드에서는 클래스가 스스로 필요한 객체를 만듭니다.

```java
public class ExampleService {
    private final ExampleRepository repository = new ExampleRepository();
}
```

이 코드에서는 `ExampleService`가 제어권을 가지고 있습니다.

- 어떤 구현체를 쓸지
- 언제 만들지
- 몇 개 만들지

를 스스로 정합니다.

그런데 Spring에서는 보통 이렇게 합니다.

```java
public class ExampleService {
    private final ExampleRepository repository;

    public ExampleService(ExampleRepository repository) {
        this.repository = repository;
    }
}
```

이제 `ExampleService`는 repository를 **스스로 만들지 않습니다.**
밖에서 넣어줍니다.

즉,

- 생성 책임
- 연결 책임

이 클래스 밖으로 나갑니다.

이렇게 **제어권이 사용자 코드에서 프레임워크/컨테이너 쪽으로 넘어가는 것**이 IoC입니다.

---

## 3-2. DI는 무엇인가

DI(Dependency Injection)는  
**의존 객체를 외부에서 주입하는 방식**입니다.

IoC는 큰 철학이고,  
DI는 IoC를 구현하는 대표적인 기술입니다.

즉:

- IoC = 큰 원리
- DI = 구체적인 구현 수단

---

## 3-3. IoC 컨테이너는 무엇인가

Spring에서는 `ApplicationContext`가 대표적인 IoC 컨테이너입니다.

이 컨테이너가 하는 일:

1. 어떤 클래스를 Bean으로 만들지 찾는다.
2. 객체를 생성한다.
3. 객체 간의 의존성을 연결한다.
4. 생명주기를 관리한다.
5. 필요하면 프록시도 만든다.
6. 싱글톤을 재사용한다.

즉,  
단순 객체 보관소가 아니라  
**애플리케이션 전체의 객체 그래프(object graph)를 조립하고 관리하는 엔진**입니다.

---

# 4. React 쪽의 "프레임워크가 제어를 가져가는 방식"

React는 Spring처럼 IoC 컨테이너라는 표현을 쓰지는 않지만,  
제어권이 프레임워크에 있다는 점에서는 IoC적인 성격이 분명합니다.

개발자는 보통 이렇게 씁니다.

- state를 선언한다
- 이벤트 핸들러를 연결한다
- JSX로 "이 상태라면 이런 화면"이라고 선언한다

그러면 React가 알아서:

- 언제 다시 렌더링할지
- 어떤 컴포넌트를 재호출할지
- 이전 virtual DOM과 무엇이 다른지 비교할지
- 실제 DOM을 어떻게 갱신할지

를 관리합니다.

즉, 개발자는 **무엇을 원한다**를 선언하고,  
React는 **언제 어떻게 적용할지**를 담당합니다.

# 5. 백엔드 파일 설명

---

## 5-1. `StudyApplication.java`

역할: Spring Boot 애플리케이션 시작점

```java
SpringApplication.run(StudyApplication.class, args);
```

이 한 줄에서 일어나는 일:

1. JVM 프로세스가 시작된다.
2. main thread가 실행된다.
3. SpringApplication이 ApplicationContext를 만든다.
4. Bean 정의(Bean Definition)를 스캔한다.
5. Bean을 heap에 생성한다.
6. 생성자 주입으로 참조를 연결한다.
7. 내장 웹서버(Tomcat)를 올린다.

즉,  
이 한 줄은 단순 run이 아니라  
**서버 + 컨테이너 + 객체 그래프 초기화**를 모두 일으킵니다.

---

## 5-2. `AppBeanConfig.java`

여기서는 `Clock` Bean을 등록했습니다.

이것은 아주 좋은 학습 포인트입니다.

많은 초보자가 "Bean은 @Service 붙은 클래스만"이라고 생각하는데, 아닙니다.

Bean 등록 방식은 크게 두 가지입니다.

1. 클래스 스캔 기반  
   - `@Component`
   - `@Service`
   - `@Repository`
   - `@Controller`

2. 명시적 설정 기반  
   - `@Configuration`
   - `@Bean`

`Clock`은 우리가 만든 클래스도 아니고, 외부 라이브러리 객체입니다.  
이런 경우 `@Bean` 방식이 자연스럽습니다.

이렇게 두는 이유:

- 시간 의존성을 고정/대체하기 쉽다.
- 테스트가 쉬워진다.
- 객체 생성 책임을 컨테이너에 맡길 수 있다.

즉, 이것도 IoC입니다.

---

## 5-3. `CorsConfig.java`

브라우저는 origin이 다른 요청을 기본적으로 막습니다.

예:

- 프론트: `http://localhost:5173`
- 백엔드: `http://localhost:8080`

포트가 다르면 origin도 다릅니다.

따라서 Spring 서버가
"이 origin의 `/api/**` 요청은 허용하겠다"
고 선언해야 합니다.

이 설정도 `WebMvcConfigurer` Bean을 등록하는 방식으로 구성했습니다.

여기서 중요한 감각:

- Spring은 "설정마저도 객체로 다룬다"
- 설정 객체 역시 컨테이너가 관리한다

---

## 5-4. `Task.java`

Entity는 데이터베이스 테이블과 연결되는 객체입니다.

초보자는 여기서 두 가지를 꼭 구분해야 합니다.

### 1) Java 객체
heap 메모리 위에 존재하는 일반 객체

### 2) DB row
영속 저장소에 존재하는 데이터

JPA는 이 둘을 매핑합니다.

즉, `Task` 객체를 수정하면  
나중에 JPA가 SQL UPDATE로 반영할 수 있습니다.

---

### JPA 영속성 컨텍스트를 감각적으로 이해하기

JPA는 단순히 SQL 실행기만이 아니라,  
Entity 객체를 추적하는 **1차 캐시/컨텍스트**를 가집니다.

이 컨텍스트는 대략:

- "현재 트랜잭션 안에서 어떤 엔티티가 로딩되었는지"
- "이 엔티티 필드가 바뀌었는지"

를 추적합니다.

따라서 어떤 시점에는

- 자바 객체 수정
- SQL 직접 호출

이 1:1로 즉시 대응되지 않습니다.

중간에 JPA가 추적하다가 flush 시점에 반영합니다.

이게 초보자에게는 약간 추상적으로 느껴지지만,  
핵심은 이겁니다.

> "메모리 안의 객체 상태와 DB 상태 사이를 JPA가 중개한다"

## 5-5. DTO들

- `TaskCreateRequest`
- `TaskUpdateRequest`
- `TaskResponse`

왜 Entity를 그대로 쓰지 않는가?

### 이유 1. 외부 입력과 내부 모델은 다르다
사용자는 `id`, `createdAt`, `updatedAt`를 보내면 안 됩니다.

### 이유 2. 검증 위치가 명확해진다
입력 DTO에 `@NotBlank`, `@Min`, `@Max`를 붙이면  
어떤 입력이 허용되는지 한눈에 보입니다.

### 이유 3. API 변경에 유연하다
DB 구조가 바뀌어도 응답 DTO는 유지할 수 있습니다.

즉, DTO는 번거로운 장식이 아니라  
**계층 경계(boundary)를 보호하는 장치**입니다.

---

## 5-6. `TaskRepository.java`

가장 중요한 파일 중 하나입니다.

인터페이스만 있고 구현체가 없습니다.

그런데 동작합니다.

왜?

Spring Data JPA가 런타임에 구현체 프록시를 만들어 Bean으로 등록하기 때문입니다.

이것은 초보자에게 매우 중요한 감각을 줍니다.

> "실무 프레임워크는 내가 클래스를 다 만들지 않아도,  
> 규약에 따라 필요한 객체를 자동 생성해준다."

즉, 이 코드도 IoC의 전형적인 사례입니다.

---

## 5-7. `TaskService.java` / `TaskServiceImpl.java`

여기가 **비즈니스 규칙의 중심**입니다.

Controller는 HTTP에 가깝고,  
Repository는 DB에 가깝습니다.

그 사이에서

- 검색 조건 정리
- 예외 발생
- 시간 처리
- Task 생성/수정/토글 로직

을 담당하는 것이 Service입니다.

### 생성자 주입을 주목하자

```java
public TaskServiceImpl(TaskRepository taskRepository, Clock clock) {
    this.taskRepository = taskRepository;
    this.clock = clock;
}
```

이 한 줄 구조가 굉장히 중요합니다.

- Service는 Repository를 직접 만들지 않습니다.
- Service는 Clock도 직접 만들지 않습니다.
- 둘 다 외부에서 들어옵니다.

즉,
Service는 "내가 누구에게 의존하는가"는 알지만,
"그 대상을 누가 어떻게 만들었는가"는 모릅니다.

이것이 바로 IoC + DI의 핵심 체감입니다.

---

### 왜 생성자 주입이 좋은가

1. 필수 의존성을 강제할 수 있다  
2. 불변 필드(`final`)를 쓸 수 있다  
3. 테스트가 쉽다  
4. 순환참조를 빨리 드러낸다  
5. 코드만 봐도 의존성이 명확하다

---

## 5-8. `TaskController.java`

Controller는 HTTP와 가장 가까운 계층입니다.

역할:

- URL 매핑
- Query Param / Path Variable / Body 받기
- 검증 시작점
- Service 호출
- JSON 응답 반환

중요한 점은  
**Controller가 똑똑해지면 안 된다**는 것입니다.

초보자는 Controller에 모든 로직을 몰아넣기 쉽습니다.

그러면:
- 테스트도 어렵고
- 책임도 흐려지고
- 재사용도 어려워집니다

Controller는 얇게 두는 것이 좋습니다.

---

## 5-9. `GlobalExceptionHandler.java`

에러는 프로젝트가 커질수록 더 중요해집니다.

이 파일이 없으면 컨트롤러마다 try-catch가 늘어날 수 있습니다.

중앙 처리의 장점:

- 에러 응답 형식 통일
- 중복 감소
- 프론트에서 처리하기 쉬움

실무에서는 이런 "공통 규칙"을 중앙에 두는 것이 매우 중요합니다.

---

## 5-10. `DataInitializer.java`

앱 시작 시 샘플 데이터를 넣습니다.

`CommandLineRunner`도 Bean이고,  
Spring Boot가 초기화 후 자동으로 `run()`을 호출합니다.

이것도 작은 IoC 예시입니다.

우리가 직접 `new DataInitializer(...).run()` 하지 않아도  
프레임워크가 적절한 타이밍에 호출합니다.

# 6. 프론트엔드 파일 설명

---

## 6-1. `main.jsx`

여기서는 React 앱의 root를 브라우저 DOM에 연결합니다.

또한 `QueryClientProvider`를 통해  
TanStack Query의 캐시 컨텍스트를 전체 앱에 공급합니다.

즉, 하위에서 `useQuery()`를 쓸 수 있는 이유는  
이 Provider가 위에 있기 때문입니다.

이 구조는 React Context 기반입니다.

조금 감각적으로 말하면:

- Spring의 ApplicationContext가 백엔드 객체 그래프를 공급하듯
- React Query는 QueryClient를 Context로 공급합니다

물론 둘은 동일한 것은 아니지만,  
"상위에서 관리 객체를 공급한다"는 감각은 비슷합니다.

---

## 6-2. `queryClient.js`

여기서 중요한 것은 **서버 상태 캐시** 개념입니다.

초보자가 흔히 하는 실수:

- 모든 것을 useState로 처리하려고 함
- 서버 데이터도 로컬 UI 데이터처럼 다루려 함

하지만 서버 데이터는 성격이 다릅니다.

### 로컬 UI 상태
- input 값
- 열림/닫힘 여부
- 현재 선택 id
- hover 여부

### 서버 상태
- 서버에서 받아온 목록
- 백엔드가 진짜 원본(source of truth)인 데이터
- stale / refetch / sync / invalidate 개념이 필요함

TanStack Query는 이 서버 상태를 위한 라이브러리입니다.

---

## 6-3. `taskApi.js`

API 호출을 한 군데에 모았습니다.

이것은 실무적으로 매우 중요합니다.

왜냐하면 컴포넌트가 직접 fetch를 남발하면:

- URL이 흩어지고
- 에러 처리 규칙도 흩어지고
- 교체 비용이 커집니다

API 레이어는 일종의 "통신 어댑터"입니다.

---

## 6-4. `useTasksQuery.js`

여기서 `useQuery`를 감각적으로 이해해야 합니다.

`useQuery`는 단순 fetch 함수가 아닙니다.

그 이상입니다.

- queryKey로 캐시 식별
- queryFn으로 데이터 획득
- 로딩 상태 제공
- 에러 상태 제공
- 캐시 저장
- refetch 관리

즉,
`useQuery`는 **fetch + cache + sync state machine**에 가깝습니다.

---

### queryKey는 왜 중요한가

`["tasks", filters]`

이 key는 "이 서버 데이터의 정체성"입니다.

다르게 말하면:

- 어떤 결과를 캐시할지
- 어떤 데이터를 무효화할지
- 어떤 데이터가 같은 요청인지

를 구분하는 주소입니다.

---

## 6-5. `useTaskMutations.js`

여기서는
- 생성
- 수정
- 토글
- 삭제

를 각각 mutation으로 관리합니다.

### 왜 useMutation인가

조회(read)는 `useQuery`
쓰기(write)는 `useMutation`

이렇게 구분하면 책임이 명확해집니다.

조회는 캐시가 중요하고,
쓰기는 성공/실패/재시도/후속 invalidate가 중요합니다.

---

### optimistic update를 꼭 이해하자

삭제나 토글 시 서버 응답을 기다리기 전에
화면을 먼저 바꾸는 전략이 optimistic update입니다.

장점:
- 사용자 반응성이 좋아짐

단점:
- 서버 실패 시 롤백 필요

따라서 이 패턴은
**빠름과 정확성 사이의 trade-off를 조절하는 기술**입니다.

---

## 6-6. `useRenderCount.js`

이 hook은 학습용으로 매우 중요합니다.

`useRef`는 렌더 사이에서 값이 유지되지만,  
값이 바뀌어도 재렌더를 일으키지 않습니다.

이것은 `useState`와 다릅니다.

### useState
- 값 변경 시 재렌더

### useRef
- 값 유지 가능
- 값 변경 시 재렌더 없음

즉, ref는
"렌더를 다시 일으킬 필요 없는 기억장소"입니다.

# 7. React Hook를 코드 수준에서 이해하기

---

## 7-1. useState

`useState`는 컴포넌트 함수가 렌더될 때마다
새 변수처럼 사라지지 않도록 React가 내부 저장소에 보관하는 state입니다.

중요한 점:

- state 값은 함수 지역변수처럼 보여도,
  실제로는 React 내부 구조에 저장됩니다.
- setter를 부르면 React가 다시 렌더링을 예약합니다.

즉, `useState`는 단순 JS 변수 저장이 아니라  
**렌더 사이를 잇는 프레임워크 관리 상태 슬롯**입니다.

---

## 7-2. useReducer

`TaskForm`에서는 reducer를 사용했습니다.

이유:
- 폼 상태가 객체 형태
- 여러 필드가 함께 움직임
- 상태 전이 규칙을 분리하고 싶음

`reducer(state, action)`는
사실상 작은 상태 머신(state machine)입니다.

```js
state + action => nextState
```

이 방식은
규모가 조금 커지는 순간 `useState` 여러 개보다 정리력이 좋아집니다.

---

## 7-3. useEffect

Effect는 "렌더 이후의 부수효과"입니다.

예:
- DOM focus
- 이벤트 리스너 등록
- 네트워크 후속 처리
- 외부 시스템 동기화

중요한 점:
React는 렌더 단계에서 순수 계산을 기대합니다.

즉:
- JSX를 반환하는 과정은 가급적 순수해야 함
- DOM 직접 조작이나 외부 세계와의 상호작용은 effect로 빼야 함

`TaskForm`에서 focus를 effect에 둔 이유가 바로 이것입니다.

---

## 7-4. useRef

`useRef`는 두 가지 용도로 자주 씁니다.

1. DOM 참조
   - input focus
2. 렌더 사이 값 저장
   - render count
   - previous value
   - mutable cache

초보자에게 자주 생기는 오해:
"ref도 state처럼 값이 바뀌면 화면이 바뀌나요?"

아니요.

ref는 React에게 "다시 그려"라고 말하지 않습니다.

즉:
- state = 렌더를 유발하는 값
- ref = 렌더를 유발하지 않는 값

---

## 7-5. useMemo

`useMemo`는 값을 메모이제이션합니다.

언제 쓸까?

- 계산 비용이 있을 때
- 참조 안정성이 중요할 때
- 하위 memo 컴포넌트와 함께 사용할 때

이번 프로젝트에서는 두 가지 용도로 사용했습니다.

1. `filters` 객체 참조 안정화
2. `stats` 계산 재사용

중요:
useMemo는 무조건 쓰는 최적화가 아닙니다.
과용하면 오히려 복잡도만 늘 수 있습니다.

---

## 7-6. useCallback

`useCallback`은 함수를 메모이제이션합니다.

왜 필요한가?

JS에서 함수는 객체입니다.
즉, 렌더할 때마다 새로 선언되면
참조가 달라집니다.

이것이 왜 문제냐면,
`React.memo`는 props를 비교할 때 참조도 봅니다.

따라서 함수 props가 매번 새로 생기면
memo된 자식도 다시 렌더될 수 있습니다.

즉:

- `React.memo`로 자식 memo
- 부모에서 `useCallback`으로 함수 참조 안정화

이 둘은 함께 쓰일 때 효과가 큽니다.

---

## 7-7. useDeferredValue

이 hook은 초보자에게 조금 낯설 수 있지만, 굉장히 실무적입니다.

검색창 예를 들어봅시다.

사용자는 타이핑을 즉시 보고 싶습니다.
하지만 그 타이핑마다 무거운 필터링/서버 요청/리렌더가 붙으면
입력 체감이 무거워질 수 있습니다.

`useDeferredValue`는
"덜 급한 갱신"을 조금 늦게 반영하여
UI 반응성을 높이는 도구입니다.

이번 프로젝트에서는
- searchInput: 즉시 바뀌는 입력값
- deferredSearchText: 조금 늦게 반영되는 값

을 분리했습니다.

---

# 8. React 최적화 핵심 개념

---

## 8-1. React.memo

`TaskRow`에 `React.memo`가 붙어 있습니다.

효과:
- props가 같으면 재렌더 생략 가능

하지만 조건이 있습니다.
props 중 함수나 객체가 매번 새로 만들어지면 효과가 떨어집니다.

그래서 최적화는 보통 세트로 갑니다.

- `React.memo`
- `useCallback`
- `useMemo`

---

## 8-2. 참조 동일성(reference equality)

초보자에게 가장 중요한 최적화 개념입니다.

JS에서 객체/배열/함수는 참조형입니다.

즉, 값이 같아 보여도

```js
{} !== {}
[] !== []
() => {} !== () => {}
```

입니다.

React memo 비교에서 이 참조가 중요합니다.

따라서 부모가 렌더될 때마다 새 객체/새 함수가 생기면  
자식 입장에서는 "props가 바뀐 것처럼" 보일 수 있습니다.

---

## 8-3. 최적화는 "무조건"이 아니다

초보자는 종종 모든 것에 useMemo/useCallback을 붙이고 싶어 합니다.

하지만 최적화에는 비용이 있습니다.

- 코드 복잡도 증가
- 가독성 저하
- 잘못 쓰면 이득이 거의 없음

좋은 기준:

1. 실제로 하위 memo 컴포넌트가 있는가?
2. 계산 비용이 의미 있게 큰가?
3. 리스트가 크거나 렌더 빈도가 높은가?

이번 예제는 "학습용이므로 의도적으로 최적화 포인트를 드러낸 것"입니다.

---

# 9. TanStack Query를 왜 써야 하는가

초보자 CRUD는 보통 이렇게 시작합니다.

```js
const [tasks, setTasks] = useState([]);
```

그리고 fetch 후 `setTasks(data)`를 합니다.

작은 프로젝트에서는 가능하지만, 조금만 커지면 문제가 생깁니다.

- 언제 다시 받아와야 하는가?
- 새로고침 시 stale 여부는?
- 여러 화면에서 같은 데이터 공유는?
- mutation 후 최신화는?
- loading / error / cache / retry는?

TanStack Query는 이 문제를 해결합니다.

즉,
로컬 UI 상태는 React state로,
서버 상태는 Query로 나누는 것이 좋습니다.

# 13. 초보자가 자주 하는 오해 정리

## 오해 1. "React state는 그냥 변수다"
아닙니다. React 내부 저장 슬롯과 연결된 값입니다.

## 오해 2. "useRef도 값이 바뀌면 화면이 바뀐다"
아닙니다. ref는 렌더를 유발하지 않습니다.

## 오해 3. "useMemo/useCallback은 무조건 좋다"
아닙니다. 필요할 때만 써야 합니다.

## 오해 4. "Repository 구현체가 없는데 어떻게 동작하나"
Spring Data JPA가 런타임 프록시를 생성합니다.

## 오해 5. "IoC는 Spring에서만 쓰는 말이다"
아닙니다. 프레임워크 일반의 공통 철학입니다.
Spring은 객체 생성/주입에서, React는 렌더링 제어에서 IoC적인 성격이 강합니다.

# 15. 최종 요약

이 프로젝트에서 반드시 가져가야 할 핵심은 다음 10개입니다.

1. Spring Boot에서 객체 생성/연결의 제어권은 IoC 컨테이너가 가진다.
2. 생성자 주입은 가장 명확하고 권장되는 DI 방식이다.
3. Controller, Service, Repository를 분리하면 책임이 선명해진다.
4. Entity와 DTO를 분리해야 경계가 보호된다.
5. React에서 로컬 UI 상태와 서버 상태는 다르게 다뤄야 한다.
6. 서버 상태는 TanStack Query가 매우 강력한 선택지다.
7. useState, useReducer, useEffect, useRef는 역할이 다 다르다.
8. useMemo와 useCallback은 참조 안정성과 비용 최적화에 사용한다.
9. React.memo는 단독보다 useCallback/useMemo와 함께 쓸 때 빛난다.
10. 좋은 풀스택 구조는 "백엔드와 프론트엔드를 각각 그 프레임워크답게" 짜는 것이다.

# 16. 프론트엔드·백엔드 설치와 빌드

이 절에서는 **처음 이 저장소를 받은 사람**이 막히지 않도록, 터미널에서 무엇을 치면 되는지까지 단계별로 정리한다.  
용어부터 짧게 짚고 간다.

- **설치(install)**: 이 프로젝트가 쓰는 **외부 라이브러리**를 내 PC로 내려받아 두는 과정이다. (백엔드는 Maven, 프론트는 npm이 담당한다.)
- **빌드(build)**: 소스 코드를 **배포 가능한 형태**(JAR, 정적 파일 폴더 등)로 묶는 과정이다.
- **실행(run / dev)**: 애플리케이션을 **켜서** 브라우저나 API 클라이언트로 확인하는 과정이다.

이 저장소는 루트 아래에 **`backend/`**(Spring Boot)와 **`frontend/`**(Vite + React)가 나란히 있다. 두 폴더는 **각각 따로** 명령을 실행한다.

## 16.2 백엔드(Spring Boot) — 설치·빌드·실행

### 16.2.1 폴더로 이동

터미널을 연 뒤, **저장소 루트**에서 `backend`로 들어간다.

```bash
cd backend
```

(Windows에서 경로가 다르면, 탐색기로 `backend` 폴더를 연 다음 주소창에 `cmd` 또는 `powershell`을 입력해 그 위치에서 터미널을 열어도 된다.)

### 16.2.2 의존성 설치 겸 빌드 — `mvn clean install`

처음 한 번은 아래를 실행한다.

```bash
mvn clean install
```

의미를 나누면 다음과 같다.

- **`clean`**: 이전 빌드 결과물(`target` 폴더 등)을 비우고 깨끗이 시작한다.
- **`install`**: 컴파일·테스트·패키징까지 하고, 로컬 Maven 저장소에 결과를 반영한다.

인터넷이 연결되어 있어야 하며, 처음에는 **시간이 꽤 걸릴 수 있다**(라이브러리를 많이 받기 때문).

테스트를 건너뛰고 싶다면(환경 문제로 테스트만 실패할 때 등):

```bash
mvn clean install -DskipTests
```

### 16.2.3 개발 중 서버 켜기 — `mvn spring-boot:run`

API와 화면을 같이 보려면 백엔드를 켠 상태가 필요하다.

```bash
mvn spring-boot:run
```

정상 기동이면 로그에 Spring 관련 메시지가 이어지고, **서버 포트는 8080**이다(`application.yml`의 `server.port`).

- **끄는 방법**: 해당 터미널에서 `Ctrl + C`.

### 16.2.4 배포용 JAR만 만들기 — `mvn package`

실행 파일(JAR)만 필요할 때:

```bash
mvn clean package -DskipTests
```

성공하면 `backend/target/` 아래에 `study-backend-0.0.1-SNAPSHOT.jar` 같은 이름의 JAR가 생긴다. 실행 예:

```bash
java -jar target/study-backend-0.0.1-SNAPSHOT.jar
```

(정확한 파일명은 `target` 폴더를 열어 확인한다.)

### 16.2.5 백엔드에서 자주 막히는 것

- **`mvn`을 찾을 수 없음**: Maven이 PATH에 없다. Maven 설치 후 터미널을 다시 연다.
- **Java 버전 오류**: JDK 21이 아닌 버전이 잡혀 있다. `java -version`으로 확인하고, IDE/터미널이 가리키는 JDK를 21로 맞춘다.
- **포트 8080 사용 중**: 다른 프로그램이 8080을 쓰고 있다. 그 프로그램을 끄거나, `application.yml`에서 포트를 바꾼 뒤 프론트 `vite.config.js`의 `proxy.target`도 같은 주소로 맞춘다.

### 16.3.1 폴더로 이동

새 터미널 탭/창을 열고(백엔드가 돌아가는 창은 그대로 두고), 저장소 루트에서:

```bash
cd frontend
```

### 16.3.2 의존성 설치 — `npm install`

**최초 1회**, 또는 `package.json`이 바뀐 뒤에 실행한다.

```bash
npm install
```

의미: `package.json`에 적힌 패키지(React, Vite, TanStack Query 등)를 **`node_modules` 폴더**로 내려받는다. 용량이 꽤 커질 수 있다.

### 16.3.3 개발 서버 켜기 — `npm run dev`

```bash
npm run dev
```

브라우저에서 **`http://localhost:5173`** 으로 접속한다. 이 프로젝트의 Vite 설정은 **5173 포트**를 쓴다.

- **끄는 방법**: `Ctrl + C`.

### 16.3.4 왜 백엔드를 먼저 켜야 할까?

개발 모드에서 프론트는 `5173`에서 돌아가고, API는 `8080`의 Spring Boot가 처리한다.  
`frontend/vite.config.js`에 **프록시**가 있어서, 브라우저는 `/api/...`로 `5173`에 요청해도 개발 서버가 **`http://localhost:8080`** 으로 넘겨준다.  
따라서 **할 일 목록 API를 쓰려면 백엔드(8080)가 켜져 있어야** 한다. 백엔드를 끈 상태면 네트워크 오류나 빈 화면에 가깝게 보일 수 있다.

### 16.3.5 배포용 정적 파일 빌드 — `npm run build`

```bash
npm run build
```
* 순수 css 파일을 사용했지만, react기반의 jsx파일은 js파일로 변환해야 한다. 
* 그래서 build가 필요하다. 
* package.json 파일안에 build = "vite build" 로 설정되어 있으므로, 실행시 npm run vite build 명령을 실행하게 된다. 

성공하면 `frontend/dist/` 폴더에 배포용 HTML/JS/CSS가 모인다.

로컬에서 빌드 결과만 빠르게 확인하려면:

```bash
npm run preview
```

(빌드 후에 실행하는 것이 일반적이다.)

### 16.3.6 프론트에서 자주 막히는 것

- **`npm`을 찾을 수 없음**: Node.js 미설치 또는 PATH 문제다.
- **`EACCES` / 권한 오류**: 전역 설치가 아니라 이 폴더에서만 `npm install`을 쓰면 대부분 해결된다. 그래도 안 되면 Node 재설치를 검토한다.
- **화면은 뜨는데 데이터가 안 온다**: 백엔드가 8080에서 실행 중인지 확인한다.

## 부록: `pom.xml`의 `<java.version>` 주석 해설

`backend/pom.xml`에 다음과 같은 주석이 있습니다.

```xml
<!-- 컴파일 타겟: Java 21 (실제 빌드 JDK는 JAVA_HOME/Toolchains가 결정) -->
<java.version>21</java.version>
```

이 주석은 두 가지를 구분하라는 의미입니다.

### "컴파일 타겟: Java 21"
- `<java.version>21</java.version>`이 하는 일
- Maven에게 "**컴파일된 `.class` 파일은 Java 21 형식이어야 한다**"고 알려줌
- 즉, **결과물(바이트코드)의 스펙**을 지정

### "실제 빌드 JDK는 JAVA_HOME/Toolchains가 결정"
- 컴파일을 **실행하는 JDK 자체**는 이 줄이 정하지 않는다는 뜻
- 그건 다음 중 하나가 결정:
  - **`JAVA_HOME`** — 셸 환경변수가 가리키는 JDK (SDKMAN, Homebrew 등이 세팅)
  - **Toolchains** — `maven-toolchains-plugin`이 설정되어 있으면 그쪽이 우선

### 비유로 보면
> "Java 21 바이트코드로 컴파일해줘"라고 **요청**하는 게 `<java.version>21</java.version>`이고,  
> 그 요청을 **수행할 JDK**(실제 일꾼)는 `JAVA_HOME` 또는 Toolchains가 데려옴.

### 실전 시나리오

| `<java.version>` | 셸 `JAVA_HOME` | 결과 |
|---|---|---|
| 21 | Java 21 | ✅ 정상 빌드 |
| 21 | Java 17 | ❌ "release version 21 not supported" — 일꾼이 21을 모름 |
| 21 | Java 25 | ✅ 정상 (Java 25는 21 바이트코드 생성 가능) |

### 왜 주석에 적어놨나
원래 주석 "Java 21을 사용한다"는 마치 이 한 줄이 모든 걸 결정하는 것처럼 읽혀서 **함정**입니다. 실제로는 `JAVA_HOME`이 다른 버전이면 빌드가 실패하는데, 처음 보는 사람은 "pom.xml에 21이라고 적었으니 21로 도는 거 아닌가?"라고 착각하기 쉽습니다. 그래서 "여기서 정하는 건 **타겟**일 뿐, **실제 JDK**는 다른 곳이 정한다"고 명시한 것입니다.

## 부록: 업계의 Java 버전 관리 표준

업계 실태는 "정답 하나"보다 **스택과 회사 규모에 따라 패턴이 갈립니다**. 2024~2026년 기준 실무 관찰입니다.

### 1) 가장 흔한 4가지 패턴

| 패턴 | 누가 주로 쓰나 | 점유율 체감 |
|---|---|---|
| **A. SDKMAN + `JAVA_HOME`** | 개인 개발자, 스타트업, 학습/사이드 프로젝트 | 매우 흔함 |
| **B. Gradle Toolchains (auto-download)** | Gradle 쓰는 모던 팀, 오픈소스 프로젝트 | **빠르게 증가 중** |
| **C. Docker로 JDK 고정 (Dockerfile에 `FROM eclipse-temurin:21`)** | 중견~대기업, 백엔드 팀, CI/CD 중심 조직 | 매우 흔함 |
| **D. `maven-toolchains-plugin`** | 레거시 다중 JDK 환경, 금융권/SI 일부 | **드묾** (설정 비용 대비 효과↓) |

### 2) 트렌드 (실무에서 실제로 일어나는 것)

#### 모던 Gradle 프로젝트 → **Gradle Toolchains가 사실상 표준**
- 이유: `build.gradle`에 5줄 적으면 끝, JDK 자동 다운로드까지 함
- Spring 공식 가이드, Micronaut, Quarkus 다 이 방식으로 문서화됨
- "내가 깔 필요도 없다"가 신입~중급 개발자에게 결정적

#### 모던 Maven 프로젝트 → **여전히 `JAVA_HOME` + (보통 SDKMAN)**
- `maven-toolchains-plugin`은 이론적으로 정답이지만 **실무에서 거의 안 씀**
- 이유: `toolchains.xml`이 머신마다 절대경로라 git에 못 올림 → "그냥 SDKMAN으로 셸 JDK 맞추는 게 낫다"는 결론
- Spring Boot 팀조차 자기 빌드는 Toolchains 안 쓰고 `JAVA_HOME`에 의존

#### 진짜 프로덕션 팀 → **Docker가 최종 정답**
- 로컬은 SDKMAN, **CI/CD와 운영은 Docker 이미지로 JDK 못박기** (`FROM eclipse-temurin:21.0.5_11-jdk-alpine`)
- 이유:
  - 빌드 재현성 100% (JDK 패치 버전까지 동일)
  - 운영 서버 JDK 패치와 빌드 JDK 패치 분리 가능
  - "내 컴퓨터에선 되는데" 문제 완전 차단
- GitHub Actions, GitLab CI, Jenkins 모두 Docker 기반 빌드가 표준

### 3) 정리: "프로 개발자"가 실제로 하는 것

회사 규모별 관찰:

**스타트업 / 1~10명 팀**
```
로컬:  SDKMAN (.sdkmanrc 커밋)
CI:    GitHub Actions의 setup-java 액션
운영:  Docker
```

**중견 / 10~100명**
```
로컬:  SDKMAN 또는 IntelliJ가 다운로드한 JDK
CI:    Docker 또는 setup-java
운영:  Docker 강제
빌드:  Gradle Toolchains (Gradle 팀) / JAVA_HOME (Maven 팀)
```

**대기업 / 100명+**
```
로컬:  사내 표준 (대개 Docker dev container, Nix, 또는 회사 배포 JDK)
CI:    Docker (이미지가 사내 레지스트리에서 옴)
운영:  Docker + Kubernetes
```

### 4) 안 쓰는 것 / 안티패턴

- ❌ **Homebrew/apt로 깐 JDK만 의존**: 머신마다 다른 마이너 버전 → "내 컴에선 되는데" 단골 원인
- ❌ **`maven-toolchains-plugin`만 단독 사용**: 설정 비용 대비 효과 적음, 신입이 못 따라옴
- ❌ **IDE(IntelliJ)가 다운로드한 JDK에만 의존**: CLI 빌드, CI, 동료와 불일치

### 5) 이 프로젝트(Spring + 학습)에 권장 순서

학습→실무 단계로 갈 때 자연스러운 진화:

1. **지금 (학습)**: SDKMAN + `.sdkmanrc` 한 줄 ✅ 이미 가는 방향
2. **팀 프로젝트로 확장**: 위에 GitHub Actions `setup-java` 추가
3. **운영 시작**: Dockerfile에 `FROM eclipse-temurin:21` 추가
4. **그 다음**: Maven은 그대로 두거나 Gradle로 옮기면 Toolchains까지

즉, **`maven-toolchains-plugin`은 건너뛰어도 됩니다.** 업계가 실제로 그쪽으로 안 갑니다. 모던 정답은 **"로컬은 SDKMAN, CI/운영은 Docker"** 의 2단 구성입니다.

### 한 줄 요약
> "프로의 정답"은 단일 도구가 아니라 **계층별 분담**: 셸은 SDKMAN, 빌드는 (선택적으로) Toolchains, **운영/CI는 Docker**.

## 18.1 Docker란 — 커널을 설치하지 않는다

"Docker는 리눅스 커널과 관련 라이브러리 파일이 설치되는 것" — **흔한 오해**입니다. 정확한 그림을 잡아둡시다.

### Docker는 리눅스 커널을 설치하지 않습니다

이게 Docker의 핵심 특징입니다.

| | VM (VirtualBox 등) | Docker |
|---|---|---|
| 리눅스 커널 | **각 VM마다 별도 커널** 설치 | **호스트 커널 공유** (커널을 안 설치함) |
| 무게 | 무거움 (GB 단위) | 가벼움 (MB 단위) |
| 부팅 | 진짜 OS 부팅 | 프로세스 시작 정도 |

**Docker 이미지에 들어있는 것은 커널이 아니라:**
- 리눅스 배포판의 **유저스페이스 파일들** (라이브러리, 명령어, 설정)
- 예: `glibc`, `bash`, `ls`, `cat` 같은 것들
- 그리고 우리 앱

**커널은 호스트 OS의 것을 빌려 씁니다** (이게 Docker가 빠른 이유).

### macOS/Windows에서는?
- macOS/Windows에는 리눅스 커널이 없음 → **Docker Desktop이 작은 리눅스 VM을 깔고**, 그 VM 안에서 컨테이너가 돔
- 즉 우리 맥에서 Docker 쓰면 사실 "맥 → 리눅스 VM → 컨테이너" 3층 구조
- 이 VM의 커널만 한 번 깔리고, 컨테이너들은 그 커널을 공유

## 18.2 호스트 OS/커널 호환성 깊이 파보기

> "호스트가 Linux가 아닌 경우를 대비해서 커널은 Docker 안에 담고 있어야 하는 것 아닌가? 그리고 Linux가 호스트라 하더라도 커널의 버전이 맞지 않는다면?"

### Q1: "커널을 Docker 안에 담으면 안 되나?"

**담을 수 없습니다.** 담는 순간 Docker가 아니라 VM이 됩니다. 이게 컨테이너의 본질입니다.

#### 컨테이너 = 호스트 커널을 공유하는 격리된 프로세스

```
[VM 방식 (VirtualBox 등)]
┌─────────────────────────────┐
│  앱                          │
│  ─────────                   │
│  유저 라이브러리             │
│  ─────────                   │
│  게스트 OS 커널 (별도!)      │  ← 무거움
│  ─────────                   │
│  가상 하드웨어 (CPU/메모리)  │
└─────────────────────────────┘
         ↑
  하이퍼바이저
         ↑
┌─────────────────────────────┐
│  호스트 OS 커널              │
└─────────────────────────────┘

[Docker 방식]
┌─────────────────────────────┐
│  앱                          │
│  ─────────                   │
│  유저 라이브러리             │
└─────────────────────────────┘
         ↑
   namespaces / cgroups (격리만)
         ↑
┌─────────────────────────────┐
│  호스트 OS 커널 (직접 공유!) │
└─────────────────────────────┘
```

#### 왜 커널을 못 담는가 — 기술적 이유

컨테이너는 **호스트 커널의 namespace/cgroup 기능으로 만들어진 격리된 프로세스 그룹**입니다. 즉:
- 컨테이너 = 호스트 커널이 만들어낸 **프로세스**
- 프로세스는 자기 안에 또 다른 커널을 돌릴 수 없음 (CPU 권한 모드가 안 됨)
- 커널을 돌리려면 → 하이퍼바이저 + 가상하드웨어 → 그게 VM

**즉, 컨테이너는 정의상 커널을 못 가집니다.** 커널을 가지면 그건 컨테이너가 아닙니다.

#### macOS/Windows에서 Docker가 하는 일 (더 정확하게)

```
[Mac에서 Docker 사용 시 실제 구조]

┌──────────────────────────────────────────┐
│  컨테이너1   컨테이너2   컨테이너3        │  ← 우리 앱들
│  ─────────────────────────────────       │
│  리눅스 유저스페이스 (각자 다른 배포판)   │
└──────────────────────────────────────────┘
                  ↑
          공유 (리눅스 namespace)
                  ↑
┌──────────────────────────────────────────┐
│  리눅스 커널 1개 (Docker Desktop의 VM)    │  ← Docker Desktop이 한 번만 설치
└──────────────────────────────────────────┘
                  ↑
       하이퍼바이저 (Apple Virtualization.framework)
                  ↑
┌──────────────────────────────────────────┐
│  macOS 커널                              │
└──────────────────────────────────────────┘
```

핵심: **이미지마다 커널이 들어있는 게 아니라**, Docker Desktop이 처음 깔릴 때 **VM과 그 안의 리눅스 커널을 한 번** 설치하고, 모든 컨테이너가 그걸 공유합니다.

→ 이미지가 가벼울 수 있는 이유 (`alpine`이 5MB로 끝나는 비결).

### Q2: "Linux 호스트라도 커널 버전이 안 맞으면?"

**훌륭한 지적입니다. 실제로 일어나는 문제입니다.** 단, Linux는 이걸 다루기 위한 강력한 규칙을 가지고 있습니다.

#### 리눅스의 철칙: "Do not break userspace"

리누스 토발즈의 유명한 원칙:
> **"커널이 바뀌어도, 옛날 프로그램은 절대 깨지면 안 된다."**

- 2.6 시절 컴파일된 바이너리가 6.x 커널에서도 동작해야 함
- 이게 30년간 강제됨
- 그래서 **대부분의 경우 컨테이너 vs 호스트 커널 버전 차이가 문제 안 됨**

#### 그래도 깨지는 경우 — 3가지 시나리오

##### (A) 새 시스템 콜을 쓰는 컨테이너 vs 옛날 호스트 커널
**가장 흔한 실제 사고**.

예시 (실제로 2021년 터진 사건):
- `glibc 2.34+`가 `faccessat2` 시스템 콜을 씀 (커널 5.8에서 추가됨)
- CentOS 7 (커널 3.10) 위에서 Ubuntu 22.04 컨테이너 실행
- → 컨테이너 안 프로그램이 `ENOSYS: Function not implemented` 에러
- → Docker가 시큐리티 필터(seccomp)에서 못 알아본 syscall 막아버린 케이스도 있음

**규칙**: **컨테이너 안 라이브러리가 쓰는 syscall이 호스트 커널에 있어야 한다.** 컨테이너는 새 커널에서 옛 컨테이너 돌리는 건 OK, **옛 커널에서 새 컨테이너 돌리는 건 위험**.

##### (B) cgroup v1 vs v2 충돌
- 컨테이너 런타임이 cgroup v2 가정 (최신 Docker)
- 호스트가 cgroup v1만 지원 (옛 RHEL 7)
- → Docker 자체가 안 뜨거나 리소스 제한이 안 먹음

##### (C) 커널 기능 의존
- `overlayfs`, `userns`, `eBPF` 같은 기능이 호스트 커널에 있어야 함
- 옛 커널엔 없을 수 있음

#### 그래서 Docker가 정한 최소 요구사항

- Docker Engine: **호스트 커널 3.10 이상** (현실적으론 4.x+ 권장)
- 그 이하는 공식적으로 지원 안 함
- 즉, Docker가 **"최소선"을 그어서 문제 영역을 좁힘**

#### 호환성 매트릭스

| 시나리오 | 동작 여부 |
|---|---|
| 옛 컨테이너 (Ubuntu 18.04) on 새 커널 (6.x) | ✅ 거의 항상 OK ("don't break userspace") |
| 새 컨테이너 (Ubuntu 24.04) on 새 커널 (6.x) | ✅ OK |
| 새 컨테이너 (Ubuntu 24.04) on 옛 커널 (3.10) | ⚠️ 종종 깨짐 (새 syscall 없음) |
| 어떤 컨테이너든 on 커널 < 3.10 | ❌ Docker 자체가 안 뜸 |

#### 실무에서 어떻게 대응하나

1. **운영 서버 커널을 최신으로 유지** — 대부분의 클라우드(EKS, GKE)는 자동으로 신선한 커널 제공
2. **Docker Desktop은 항상 VM 안 커널을 최신으로 유지** → macOS/Windows 사용자는 사실상 이 문제 안 겪음
3. **베이스 이미지의 `glibc` 버전을 신경 쓴다** — Alpine은 `musl libc`라 또 다른 호환성 매트릭스
4. **CI에서 운영과 같은 커널로 테스트**

#### 그래도 안 되면 — 결국 VM이 답
정말 다른 커널이 필요하다면 (예: Windows 컨테이너를 Linux 호스트에서, 또는 옛 커널만 도는 레거시 앱)
→ **컨테이너 안 됨, VM 써야 함**
→ Kata Containers, Firecracker 같은 "VM처럼 동작하는 컨테이너"가 그 자리를 채움

### Q1·Q2 정리

#### Q1: 왜 커널을 안 담나?
**담는 순간 컨테이너가 아니라 VM**이라서. 컨테이너 = 호스트 커널 공유가 정의.

#### Q2: Linux에서 커널 버전 안 맞으면?
**대부분 OK** (리누스의 "don't break userspace" 철칙) — 단:
- 옛 커널 + 새 컨테이너 = ⚠️ 새 syscall 사용 시 깨짐
- 너무 옛 커널 (<3.10) = ❌ Docker 자체가 안 뜸
- 그래서 **운영 서버 커널을 최신으로 유지**가 실무 룰

#### 한 줄 정리
> Docker는 **"호스트 커널을 신뢰한다"는 전제 위에 세워진 도구**입니다.  
> 그 전제가 깨질 만큼 옛날 커널이면 Docker가 거부하고, 너무 새 syscall을 쓰면 컨테이너가 깨집니다 — 그 사이의 넓은 안전 구간이 우리가 평소 쓰는 영역입니다.

## 18.3 Docker 설치와 이미지 안 라이브러리 설치

### 설치 명령은 두 종류가 있습니다

#### (A) Docker 자체를 설치 (호스트에 한 번)
```bash
# macOS (Homebrew)
brew install --cask docker

# 또는 공식 사이트에서 Docker Desktop dmg 다운로드
# Linux (Ubuntu)
sudo apt install docker.io

# 설치 확인
docker --version
docker run hello-world   # 동작 테스트
```

#### (B) 이미지 안에 라이브러리 설치 (Dockerfile에서)
이게 진짜 "라이브러리 설치 명령"입니다. **`RUN` 명령**으로 베이스 이미지의 패키지 매니저를 호출:

```dockerfile
# Debian/Ubuntu 계열 베이스
FROM eclipse-temurin:21-jdk
RUN apt-get update && apt-get install -y curl git
```

```dockerfile
# Alpine 계열 베이스 (훨씬 가벼움)
FROM eclipse-temurin:21-jdk-alpine
RUN apk add --no-cache curl git
```

```dockerfile
# Red Hat 계열
FROM registry.access.redhat.com/ubi9/openjdk-21
RUN dnf install -y curl git
```

베이스 이미지에 따라 **패키지 매니저가 다릅니다**:

| 베이스 이미지 | 배포판 | 패키지 매니저 |
|---|---|---|
| `eclipse-temurin:21-jdk` | Ubuntu | `apt-get` |
| `eclipse-temurin:21-jdk-alpine` | Alpine | `apk` |
| `eclipse-temurin:21-jdk-jammy` | Ubuntu 22.04 | `apt-get` |
| `ubi9/openjdk-21` | RHEL UBI | `dnf` |

## 18.5 Docker와 pom.xml의 관계 — "Docker로 가면 pom.xml은 필요없는가?"

**아니요, pom.xml은 여전히 필수입니다. 오히려 더 중요해집니다.** 둘은 **서로 다른 문제를 해결**합니다.

### 핵심 구분

| 도구 | 역할 | 비유 |
|---|---|---|
| **Dockerfile** | **환경** 제공 (OS + JDK + 도구) | 주방 |
| **pom.xml** | **레시피** 정의 (의존성, 빌드 순서, 산출물) | 요리법 |

주방(Docker)이 아무리 좋아도 **요리법(pom.xml)이 없으면 뭘 만들지 모릅니다.** Docker는 단지 `mvn package` 명령을 실행할 환경을 제공할 뿐이고, 그 `mvn` 명령은 여전히 pom.xml을 읽어야 합니다.

### Docker 환경에서 pom.xml이 하는 일

pom.xml이 없으면 Docker 안에서 다음을 **아무도 모릅니다**:
- Spring Boot 3.5.14를 받아 와야 한다는 사실
- JPA, Web, Validation 의존성 목록
- 어떤 메인 클래스를 실행할 JAR로 패키징할지
- groupId / artifactId / version

Docker는 이 중 **아무것도 하지 않습니다.** Docker는 그냥 "Java 21이 깔린 리눅스"를 던져줄 뿐입니다.

### Docker가 바꾸는 것 (vs 바꾸지 않는 것)

**바뀌는 것**:
- 로컬 JDK 설치 여부가 빌드에 영향 없음
- JDK 패치 버전까지 완벽히 고정 (`21.0.5_11`처럼 명시)
- "JAVA_HOME" 논쟁이 사라짐 (컨테이너 안에 하나뿐)
- CI/운영 환경 완전 동일

**안 바뀌는 것** (= pom.xml의 영역):
- 의존성 관리
- 빌드 설정 (Spring Boot 플러그인, 컴파일러 옵션)
- `<java.version>21</java.version>` — 여전히 필요
- 산출물 메타데이터

### `<java.version>21</java.version>`은 Docker 시대에 어떻게 되나?

여전히 필요합니다. 의미가 약간 바뀔 뿐입니다:

| 환경 | `<java.version>21</java.version>`의 역할 |
|---|---|
| Docker 없이 | "요청"만 함, 실제 JDK는 `JAVA_HOME`이 결정 (불일치 위험 ⚠️) |
| Docker 사용 | "요청"이 곧 보장됨, 컨테이너 JDK가 21로 못박혀 있어 항상 일치 ✅ |

즉, Docker는 `<java.version>`을 **없애는 게 아니라 "거짓말이 될 수 없게 만드는"** 역할입니다.

### 단 한 가지 예외: Gradle + Toolchains는 pom.xml 자체가 없음
- Gradle 프로젝트는 `build.gradle`이 pom.xml의 역할 대신함
- Docker로 가도 `build.gradle`은 필요
- 즉 "빌드 설정 파일" 자체는 어떤 빌드 도구든 필요

### 정리

> **Docker는 "JDK를 어디서 가져오나"를 해결하고,**  
> **pom.xml은 "무엇을 빌드하나"를 해결합니다.**  
> 둘은 대체관계가 아니라 **보완관계**입니다.

Docker 기반으로 가도 pom.xml은 **그대로 두면 됩니다** — 오히려 Docker가 pom.xml의 선언(`<java.version>21`)을 **실제로 보장해주는 파트너**가 됩니다.

## 18.6 우리 프로젝트의 Dockerfile

### Maven Wrapper 도입에 따른 베이스 이미지 변경

| 항목 | Before | After |
|---|---|---|
| 빌드 베이스 | `maven:3.9-eclipse-temurin-21-alpine` (Maven 포함, 무거움) | **`eclipse-temurin:21-jdk-alpine`** (Maven 빼고 더 가벼움) |
| Maven 출처 | 베이스 이미지에 들어있는 것 | **`mvnw`가 다운로드한 3.9.16** (git에 박힌 버전) |
| 빌드 명령 | `mvn dependency:go-offline` | `./mvnw dependency:go-offline` |

### `backend/Dockerfile` 전체

```dockerfile
# =========================================================
# Stage 1: 빌드 (JDK 21 + Maven Wrapper로 컴파일/패키징)
#   - Maven은 mvnw가 자동으로 다운로드하므로 베이스 이미지는 JDK만 있으면 됨
#   - pom.xml만 먼저 복사해 의존성 캐시 레이어를 분리
# =========================================================
FROM eclipse-temurin:21-jdk-alpine AS build
WORKDIR /app

# 1) Maven Wrapper와 pom.xml 먼저 복사 (의존성 캐시 활용)
COPY mvnw .
COPY .mvn .mvn
COPY pom.xml .
RUN ./mvnw -B -q dependency:go-offline

# 2) 소스 복사 후 패키징 (테스트는 CI에서 따로 돌린다고 가정)
COPY src ./src
RUN ./mvnw -B -q package -DskipTests

# =========================================================
# Stage 2: 실행 (JRE만 들고 작은 이미지로)
#   - JDK 대신 JRE 사용 → 이미지 크기 절감
#   - non-root 사용자로 실행 (보안 모범사례)
# =========================================================
FROM eclipse-temurin:21-jre-alpine
WORKDIR /app

# non-root 사용자 생성 후 전환
RUN addgroup -S spring && adduser -S spring -G spring
USER spring:spring

# 빌드 산출물 JAR만 복사
COPY --from=build /app/target/study-backend-0.0.1-SNAPSHOT.jar app.jar

# Spring Boot 기본 포트
EXPOSE 8080

# 컨테이너 시작 명령
ENTRYPOINT ["java", "-jar", "/app/app.jar"]
```

### Dockerfile 설계 결정 요약

| 결정 | 이유 |
|---|---|
| `eclipse-temurin:21-jdk-alpine` 빌드 베이스 | Maven Wrapper가 Maven을 가져오므로 JDK만 있으면 충분 |
| 멀티스테이지 (build → runtime) | 최종 이미지에 Maven/JDK 안 들어감 → ~150MB로 축소 |
| `pom.xml` 먼저 복사 후 `dependency:go-offline` | 소스만 바뀔 땐 의존성 캐시 재사용 → 빌드 빨라짐 |
| `21-jre-alpine` 런타임 | JDK 대신 JRE만, Alpine → 가장 가벼움 |
| `non-root` 사용자 (`spring`) | 컨테이너 보안 모범사례 |
| `study-backend-0.0.1-SNAPSHOT.jar` 직접 명시 | pom.xml의 artifactId-version과 일치 |

### 검증
```bash
cd backend
docker build -t study-backend:latest .       # 빌드
docker run --rm -p 8080:8080 study-backend   # 실행
```

이렇게 하면 **로컬 빌드든 Docker 빌드든 CI든 모두 동일한 Maven 3.9.16 + Java 21**로 동작합니다. 빌드 재현성이 한 계단 더 강해졌습니다.

### 추가 권장 (다음 단계)
- **`.dockerignore`**: `target/`, `.idea/`, `.DS_Store` 등을 빌드 컨텍스트에서 제외 → 빌드 속도/이미지 크기 개선
- **`docker-compose.yml`**: 프론트엔드(React) + 백엔드를 한 번에 띄우기
- **헬스체크**: Spring Actuator로 `HEALTHCHECK` 추가

## 18.7 `docker build`의 결과물 — JAR은 어디에 생기나

> "docker build로 빌드하면 JAR 실행파일이 생성되는가?"

**답: JAR 파일은 호스트에 생기지 않습니다.** 다만 **이미지 안에는 있습니다.** 이 구분이 중요합니다.

### `mvn package` vs `docker build` — 결과물이 다릅니다

| 명령 | 결과물 | 어디에 생기나 |
|---|---|---|
| `./mvnw package` | **JAR 파일** | 호스트 파일시스템의 `target/study-backend-0.0.1-SNAPSHOT.jar` |
| `docker build -t study-backend .` | **Docker 이미지** | Docker가 관리하는 내부 저장소 (호스트 파일시스템 아님) |

`docker build`는 **JAR 파일을 만들지 않습니다.** "JAR이 들어있는 이미지"를 만듭니다.

### 우리 Dockerfile에서 일어나는 일

```dockerfile
# Stage 1 (build): 컨테이너 안에서 JAR 생성
FROM eclipse-temurin:21-jdk-alpine AS build
RUN ./mvnw -B -q package -DskipTests
# → 이 시점에 컨테이너 안에 /app/target/study-backend-0.0.1-SNAPSHOT.jar 생성됨
# → 하지만 이 컨테이너는 일시적이고, 호스트엔 파일이 안 보임

# Stage 2 (runtime): Stage 1의 JAR을 새 이미지로 복사
FROM eclipse-temurin:21-jre-alpine
COPY --from=build /app/target/study-backend-0.0.1-SNAPSHOT.jar app.jar
# → 최종 이미지 안에 /app/app.jar로 들어감
```

빌드 후 `backend/target/` 디렉토리를 봐도 **새 JAR이 안 생깁니다.** (이전에 `./mvnw`로 직접 빌드한 적 있다면 그 잔재가 있을 뿐)

### 확인 방법

#### 1) 이미지가 생겼는지 확인
```bash
docker images | grep study-backend
# study-backend   latest   a1b2c3...   2 minutes ago   165MB
```

#### 2) 이미지 안의 JAR 직접 확인
```bash
docker run --rm study-backend ls -lh /app/
# -rw-r--r-- 1 spring spring 48M Jun 12 13:50 app.jar
```

#### 3) 이미지 안의 JAR을 호스트로 꺼내기 (필요하다면)
```bash
# 방법 A: 컨테이너 만들고 docker cp
docker create --name temp study-backend
docker cp temp:/app/app.jar ./extracted.jar
docker rm temp

# 방법 B: 한 줄로
docker run --rm -v $(pwd):/out --entrypoint cp study-backend /app/app.jar /out/extracted.jar
```

### 실행 방법

JAR 파일을 꺼낼 필요 없이 **이미지 자체를 실행**하면 됩니다:

```bash
docker run --rm -p 8080:8080 study-backend
# → 컨테이너 시작 → 내부적으로 java -jar /app/app.jar 실행
# → 호스트의 8080 포트로 Spring Boot 접근 가능
```

### 그림으로 정리

```
┌─────────────────────────────────────────────────────────────┐
│  방식 A: ./mvnw package                                      │
│                                                             │
│  소스코드 ──→ [Maven] ──→ target/*.jar  (호스트에 파일)      │
│                          ↓                                  │
│                       java -jar로 실행                       │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│  방식 B: docker build                                        │
│                                                             │
│  소스코드 ──→ [컨테이너 안에서 Maven 빌드]                   │
│                ↓                                            │
│              JAR이 컨테이너 안에서 생성                       │
│                ↓                                            │
│              최종 이미지로 패키징                             │
│                                                             │
│  결과물: Docker 이미지 (이미지 안에 JAR 들어있음)              │
│  실행:   docker run → 이미지에서 컨테이너 생성 → java 실행     │
└─────────────────────────────────────────────────────────────┘
```

### 왜 이렇게 만드나 — Docker의 철학

JAR을 호스트에 안 두는 게 **의도**입니다. 이유:
- **격리**: 호스트 환경 오염 안 시킴 (Java 깔려있지 않아도 OK)
- **이식성**: 이미지 하나만 옮기면 어디서든 동일하게 실행
- **재현성**: "내 컴에 있는 그 JAR" 같은 모호함이 사라짐
- **배포**: `docker push` → 레지스트리 → `docker pull`로 그대로 배포

### 한 줄 요약
> `docker build` → **JAR이 아닌 "JAR을 품은 이미지"가 결과물**.  
> JAR을 따로 꺼낼 필요 없이 `docker run`으로 이미지를 실행하면 됩니다.

## 18.8 Docker 한 줄 요약

> **Docker는 호스트 커널을 공유하는 격리된 프로세스 환경**이며,  
> **pom.xml(빌드 레시피)과 Dockerfile(빌드/실행 환경)이 보완 관계**로 동작해  
> **"내 컴퓨터에선 되는데" 문제를 원천 차단**합니다.

## 부록: Maven Wrapper (mvnw)란?

**"프로젝트 안에 Maven 자체를 끼워 넣는 장치"** 입니다. 누가 받든 머신에 Maven이 깔려있지 않아도 같은 버전의 Maven으로 빌드할 수 있게 해줍니다.

### 1) 핵심 아이디어

| 일반 Maven | Maven Wrapper |
|---|---|
| 머신에 `mvn` 설치 필요 | **설치 불필요** |
| `mvn clean package` | `./mvnw clean package` |
| 머신마다 Maven 버전 다를 수 있음 | **모든 개발자가 같은 버전** |
| 신규 입사자가 `brew install maven` 해야 함 | git clone → 바로 빌드 OK |

### 2) 파일 구성 (4개)

`mvn wrapper:wrapper` 명령을 한 번 실행하면 프로젝트 루트에 다음이 생깁니다:

```
backend/
├── mvnw                          # macOS/Linux용 스크립트 (실행 가능)
├── mvnw.cmd                      # Windows용 배치 파일
├── .mvn/
│   └── wrapper/
│       └── maven-wrapper.properties   # 어느 Maven 버전 쓸지 명시
└── pom.xml
```

### 3) `maven-wrapper.properties`의 정체

```properties
wrapperVersion=3.3.4
distributionType=only-script
distributionUrl=https://repo.maven.apache.org/maven2/org/apache/maven/apache-maven/3.9.16/apache-maven-3.9.16-bin.zip
```

이게 핵심입니다 — **"Maven 3.9.16을 여기서 받아 써라"** 라고 못박는 것. 처음 `./mvnw` 실행 시 이 URL에서 Maven을 다운로드해 `~/.m2/wrapper/`에 캐싱하고, 그 이후로는 그 Maven을 사용합니다.

### 4) 어떻게 동작하나

```bash
./mvnw clean package
```
↓
1. `mvnw` 스크립트가 `.mvn/wrapper/maven-wrapper.properties` 읽음
2. 지정된 Maven 버전이 캐시에 있는지 확인
3. 없으면 다운로드 (한 번만)
4. 캐싱된 Maven으로 빌드 실행

### 5) 왜 만들었나 — 해결하는 문제

**"내 컴엔 Maven 3.6, 너는 3.9, CI는 3.8" 같은 버전 불일치 문제**

- 어떤 사람: `brew install maven` → 3.9
- 다른 사람: 회사 표준 이미지 → 3.6
- CI 서버: 또 다른 버전
- → 가끔 미묘한 빌드 차이 발생

Maven Wrapper는 **Maven 버전을 git에 같이 커밋**해서 이 문제를 원천 차단합니다.

### 6) Gradle도 똑같은 게 있다

- Gradle Wrapper (`gradlew`, `gradlew.bat`)
- 사실 **Maven Wrapper가 Gradle Wrapper를 따라 만든** 것 (2018년 정식 도입)
- Gradle은 처음부터 Wrapper 표준이라 더 일반적

### 7) 업계 관행

- **거의 모든 모던 Spring Boot 프로젝트는 Maven/Gradle Wrapper 포함이 표준**
- Spring Initializr(start.spring.io)로 새 프로젝트 만들면 자동으로 `mvnw`가 같이 옵니다
- 학습 프로젝트라도 추가해두면 좋은 습관

### 8) 우리 프로젝트에 실제로 적용한 결과

#### 작업 1: `pom.xml` 오타 수정
`</parent>` 뒤에 한글 "ㄹ" 한 글자가 끼어있어 Maven이 XML 파싱을 못하던 것을 발견 → 제거.

#### 작업 2: Wrapper 생성
```bash
cd backend
mvn wrapper:wrapper
```
결과:
```
backend/
├── mvnw                          (12KB, 실행권한 있음)
├── mvnw.cmd                      (8.3KB, Windows용)
└── .mvn/wrapper/
    └── maven-wrapper.properties  (Maven 3.9.16 못박음)
```
`./mvnw -v` 동작 확인 → Maven 3.9.16 + Java 21로 정상 동작.

> Wrapper 도입에 따른 **Dockerfile 갱신**은 "18. Docker 설정" 섹션 참조.

### 9) 검증 명령
```bash
cd backend
./mvnw clean package -DskipTests        # 로컬 빌드 확인
```

### 10) 한 줄 요약
> Maven Wrapper = **Maven을 git에 커밋해서 "이 프로젝트는 정확히 이 Maven 버전으로 빌드한다"고 못박는 장치**.
