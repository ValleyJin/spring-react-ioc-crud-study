# 강의.md
## Spring Boot IoC + React Hooks + TanStack Query CRUD + React 최적화 통합 자습서

이 문서는 단순한 README가 아니라,  
**왜 이 코드가 이렇게 생겼는지**를  
**메모리 관점**, **프레임워크 관점**, **초보자 관점**, **실무 뼈대 관점**에서 설명하는 강의 문서입니다.

---

# 0. 이 프로젝트의 학습 목표

이 프로젝트 하나로 아래 4가지를 연결해서 이해하는 것이 목표입니다.

1. **Spring Boot IoC / DI / Bean**
2. **React의 상태 관리와 Hook**
3. **TanStack Query를 이용한 서버 상태 관리**
4. **React 렌더링 최적화의 실제 감각**

즉,

- 백엔드에서 객체를 누가 만들고 연결하는가?
- 프론트엔드에서 state가 바뀌면 화면은 어떻게 다시 그려지는가?
- 서버 데이터와 로컬 UI 데이터는 왜 다르게 다뤄야 하는가?
- 함수와 객체의 참조가 왜 렌더링 성능에 영향을 주는가?

이 질문에 대한 감각을 얻는 것이 핵심입니다.

# 1. 전체 구조를 먼저 보자

```text
브라우저
  ↓
Vite Dev Server / React App
  ↓ (fetch / TanStack Query)
Spring Boot Controller
  ↓
Service
  ↓
Repository
  ↓
H2 Database
```

조금 더 세밀하게 보면:

```text
[React Component]
  ├─ local UI state (useState, useReducer)
  ├─ server state access (useQuery, useMutation)
  ├─ derived state (useMemo)
  └─ optimized callbacks (useCallback, React.memo)
           ↓
        /api/tasks
           ↓
[TaskController]
           ↓
[TaskServiceImpl]
           ↓
[TaskRepository]
           ↓
[H2 / JPA]
```

# 2. 왜 이 프로젝트가 좋은 뼈대인가

초보자는 종종 이렇게 배웁니다.

- React 따로
- Spring 따로
- fetch 따로
- Hook 따로
- 최적화 따로

이렇게 배우면 개념은 조각나는데, **실제 서비스 한 덩어리**로 연결되는 감각이 잘 안 생깁니다.

이 프로젝트는 그 조각들을 한 번에 연결합니다.

- Spring에서는 **IoC 컨테이너가 객체를 생성/주입**
- React에서는 **Hook이 렌더링 생명주기 안에서 state/side effect를 관리**
- TanStack Query는 **서버 상태를 캐시하고 동기화**
- React.memo, useMemo, useCallback은 **참조 안정성과 불필요한 재계산/재렌더를 줄임**

즉,  
이 프로젝트는 단순 CRUD 이상으로  
**프레임워크가 개발자의 제어를 어떻게 가져가는지**를 보여줍니다.

---

# 3. IoC, DI, Container를 가장 쉽게 이해하기

## 3-1. IoC란 무엇인가

IoC(Inversion of Control)는 직역하면 **제어의 역전**입니다.

보통 객체지향 초급 코드에서는 클래스가 스스로 필요한 객체를 만듭니다.

```java
public class ExampleService {
    private final ExampleRepository repository = new ExampleRepository();
}
```

이 코드에서는 `ExampleService`가 제어권을 가지고 있습니다.

- 어떤 구현체를 쓸지
- 언제 만들지
- 몇 개 만들지

를 스스로 정합니다.

그런데 Spring에서는 보통 이렇게 합니다.

```java
public class ExampleService {
    private final ExampleRepository repository;

    public ExampleService(ExampleRepository repository) {
        this.repository = repository;
    }
}
```

이제 `ExampleService`는 repository를 **스스로 만들지 않습니다.**
밖에서 넣어줍니다.

즉,

- 생성 책임
- 연결 책임

이 클래스 밖으로 나갑니다.

이렇게 **제어권이 사용자 코드에서 프레임워크/컨테이너 쪽으로 넘어가는 것**이 IoC입니다.

---

## 3-2. DI는 무엇인가

DI(Dependency Injection)는  
**의존 객체를 외부에서 주입하는 방식**입니다.

IoC는 큰 철학이고,  
DI는 IoC를 구현하는 대표적인 기술입니다.

즉:

- IoC = 큰 원리
- DI = 구체적인 구현 수단

---

## 3-3. IoC 컨테이너는 무엇인가

Spring에서는 `ApplicationContext`가 대표적인 IoC 컨테이너입니다.

이 컨테이너가 하는 일:

1. 어떤 클래스를 Bean으로 만들지 찾는다.
2. 객체를 생성한다.
3. 객체 간의 의존성을 연결한다.
4. 생명주기를 관리한다.
5. 필요하면 프록시도 만든다.
6. 싱글톤을 재사용한다.

즉,  
단순 객체 보관소가 아니라  
**애플리케이션 전체의 객체 그래프(object graph)를 조립하고 관리하는 엔진**입니다.

---

# 4. React 쪽의 "프레임워크가 제어를 가져가는 방식"

React는 Spring처럼 IoC 컨테이너라는 표현을 쓰지는 않지만,  
제어권이 프레임워크에 있다는 점에서는 IoC적인 성격이 분명합니다.

개발자는 보통 이렇게 씁니다.

- state를 선언한다
- 이벤트 핸들러를 연결한다
- JSX로 "이 상태라면 이런 화면"이라고 선언한다

그러면 React가 알아서:

- 언제 다시 렌더링할지
- 어떤 컴포넌트를 재호출할지
- 이전 virtual DOM과 무엇이 다른지 비교할지
- 실제 DOM을 어떻게 갱신할지

를 관리합니다.

즉, 개발자는 **무엇을 원한다**를 선언하고,  
React는 **언제 어떻게 적용할지**를 담당합니다.

# 5. 백엔드 파일 설명

---

## 5-1. `StudyApplication.java`

역할: Spring Boot 애플리케이션 시작점

```java
SpringApplication.run(StudyApplication.class, args);
```

이 한 줄에서 일어나는 일:

1. JVM 프로세스가 시작된다.
2. main thread가 실행된다.
3. SpringApplication이 ApplicationContext를 만든다.
4. Bean 정의를 스캔한다.
5. Bean을 heap에 생성한다.
6. 생성자 주입으로 참조를 연결한다.
7. 내장 웹서버(Tomcat)를 올린다.

즉,  
이 한 줄은 단순 run이 아니라  
**서버 + 컨테이너 + 객체 그래프 초기화**를 모두 일으킵니다.

---

## 5-2. `AppBeanConfig.java`

여기서는 `Clock` Bean을 등록했습니다.

이것은 아주 좋은 학습 포인트입니다.

많은 초보자가 "Bean은 @Service 붙은 클래스만"이라고 생각하는데, 아닙니다.

Bean 등록 방식은 크게 두 가지입니다.

1. 클래스 스캔 기반  
   - `@Component`
   - `@Service`
   - `@Repository`
   - `@Controller`

2. 명시적 설정 기반  
   - `@Configuration`
   - `@Bean`

`Clock`은 우리가 만든 클래스도 아니고, 외부 라이브러리 객체입니다.  
이런 경우 `@Bean` 방식이 자연스럽습니다.

이렇게 두는 이유:

- 시간 의존성을 고정/대체하기 쉽다.
- 테스트가 쉬워진다.
- 객체 생성 책임을 컨테이너에 맡길 수 있다.

즉, 이것도 IoC입니다.

---

## 5-3. `CorsConfig.java`

브라우저는 origin이 다른 요청을 기본적으로 막습니다.

예:

- 프론트: `http://localhost:5173`
- 백엔드: `http://localhost:8080`

포트가 다르면 origin도 다릅니다.

따라서 Spring 서버가
"이 origin의 `/api/**` 요청은 허용하겠다"
고 선언해야 합니다.

이 설정도 `WebMvcConfigurer` Bean을 등록하는 방식으로 구성했습니다.

여기서 중요한 감각:

- Spring은 "설정마저도 객체로 다룬다"
- 설정 객체 역시 컨테이너가 관리한다

---

## 5-4. `Task.java`

Entity는 데이터베이스 테이블과 연결되는 객체입니다.

초보자는 여기서 두 가지를 꼭 구분해야 합니다.

### 1) Java 객체
heap 메모리 위에 존재하는 일반 객체

### 2) DB row
영속 저장소에 존재하는 데이터

JPA는 이 둘을 매핑합니다.

즉, `Task` 객체를 수정하면  
나중에 JPA가 SQL UPDATE로 반영할 수 있습니다.

---

### JPA 영속성 컨텍스트를 감각적으로 이해하기

JPA는 단순히 SQL 실행기만이 아니라,  
Entity 객체를 추적하는 **1차 캐시/컨텍스트**를 가집니다.

이 컨텍스트는 대략:

- "현재 트랜잭션 안에서 어떤 엔티티가 로딩되었는지"
- "이 엔티티 필드가 바뀌었는지"

를 추적합니다.

따라서 어떤 시점에는

- 자바 객체 수정
- SQL 직접 호출

이 1:1로 즉시 대응되지 않습니다.

중간에 JPA가 추적하다가 flush 시점에 반영합니다.

이게 초보자에게는 약간 추상적으로 느껴지지만,  
핵심은 이겁니다.

> "메모리 안의 객체 상태와 DB 상태 사이를 JPA가 중개한다"

---

## 5-5. DTO들

- `TaskCreateRequest`
- `TaskUpdateRequest`
- `TaskResponse`

왜 Entity를 그대로 쓰지 않는가?

### 이유 1. 외부 입력과 내부 모델은 다르다
사용자는 `id`, `createdAt`, `updatedAt`를 보내면 안 됩니다.

### 이유 2. 검증 위치가 명확해진다
입력 DTO에 `@NotBlank`, `@Min`, `@Max`를 붙이면  
어떤 입력이 허용되는지 한눈에 보입니다.

### 이유 3. API 변경에 유연하다
DB 구조가 바뀌어도 응답 DTO는 유지할 수 있습니다.

즉, DTO는 번거로운 장식이 아니라  
**계층 경계(boundary)를 보호하는 장치**입니다.

---

## 5-6. `TaskRepository.java`

가장 중요한 파일 중 하나입니다.

인터페이스만 있고 구현체가 없습니다.

그런데 동작합니다.

왜?

Spring Data JPA가 런타임에 구현체 프록시를 만들어 Bean으로 등록하기 때문입니다.

이것은 초보자에게 매우 중요한 감각을 줍니다.

> "실무 프레임워크는 내가 클래스를 다 만들지 않아도,  
> 규약에 따라 필요한 객체를 자동 생성해준다."

즉, 이 코드도 IoC의 전형적인 사례입니다.

---

## 5-7. `TaskService.java` / `TaskServiceImpl.java`

여기가 **비즈니스 규칙의 중심**입니다.

Controller는 HTTP에 가깝고,  
Repository는 DB에 가깝습니다.

그 사이에서

- 검색 조건 정리
- 예외 발생
- 시간 처리
- Task 생성/수정/토글 로직

을 담당하는 것이 Service입니다.

### 생성자 주입을 주목하자

```java
public TaskServiceImpl(TaskRepository taskRepository, Clock clock) {
    this.taskRepository = taskRepository;
    this.clock = clock;
}
```

이 한 줄 구조가 굉장히 중요합니다.

- Service는 Repository를 직접 만들지 않습니다.
- Service는 Clock도 직접 만들지 않습니다.
- 둘 다 외부에서 들어옵니다.

즉,
Service는 "내가 누구에게 의존하는가"는 알지만,
"그 대상을 누가 어떻게 만들었는가"는 모릅니다.

이것이 바로 IoC + DI의 핵심 체감입니다.

---

### 왜 생성자 주입이 좋은가

1. 필수 의존성을 강제할 수 있다  
2. 불변 필드(`final`)를 쓸 수 있다  
3. 테스트가 쉽다  
4. 순환참조를 빨리 드러낸다  
5. 코드만 봐도 의존성이 명확하다

---

## 5-8. `TaskController.java`

Controller는 HTTP와 가장 가까운 계층입니다.

역할:

- URL 매핑
- Query Param / Path Variable / Body 받기
- 검증 시작점
- Service 호출
- JSON 응답 반환

중요한 점은  
**Controller가 똑똑해지면 안 된다**는 것입니다.

초보자는 Controller에 모든 로직을 몰아넣기 쉽습니다.

그러면:
- 테스트도 어렵고
- 책임도 흐려지고
- 재사용도 어려워집니다

Controller는 얇게 두는 것이 좋습니다.

---

## 5-9. `GlobalExceptionHandler.java`

에러는 프로젝트가 커질수록 더 중요해집니다.

이 파일이 없으면 컨트롤러마다 try-catch가 늘어날 수 있습니다.

중앙 처리의 장점:

- 에러 응답 형식 통일
- 중복 감소
- 프론트에서 처리하기 쉬움

실무에서는 이런 "공통 규칙"을 중앙에 두는 것이 매우 중요합니다.

---

## 5-10. `DataInitializer.java`

앱 시작 시 샘플 데이터를 넣습니다.

`CommandLineRunner`도 Bean이고,  
Spring Boot가 초기화 후 자동으로 `run()`을 호출합니다.

이것도 작은 IoC 예시입니다.

우리가 직접 `new DataInitializer(...).run()` 하지 않아도  
프레임워크가 적절한 타이밍에 호출합니다.

# 6. 프론트엔드 파일 설명

---

## 6-1. `main.jsx`

여기서는 React 앱의 root를 브라우저 DOM에 연결합니다.

또한 `QueryClientProvider`를 통해  
TanStack Query의 캐시 컨텍스트를 전체 앱에 공급합니다.

즉, 하위에서 `useQuery()`를 쓸 수 있는 이유는  
이 Provider가 위에 있기 때문입니다.

이 구조는 React Context 기반입니다.

조금 감각적으로 말하면:

- Spring의 ApplicationContext가 백엔드 객체 그래프를 공급하듯
- React Query는 QueryClient를 Context로 공급합니다

물론 둘은 동일한 것은 아니지만,  
"상위에서 관리 객체를 공급한다"는 감각은 비슷합니다.

---

## 6-2. `queryClient.js`

여기서 중요한 것은 **서버 상태 캐시** 개념입니다.

초보자가 흔히 하는 실수:

- 모든 것을 useState로 처리하려고 함
- 서버 데이터도 로컬 UI 데이터처럼 다루려 함

하지만 서버 데이터는 성격이 다릅니다.

### 로컬 UI 상태
- input 값
- 열림/닫힘 여부
- 현재 선택 id
- hover 여부

### 서버 상태
- 서버에서 받아온 목록
- 백엔드가 진짜 원본(source of truth)인 데이터
- stale / refetch / sync / invalidate 개념이 필요함

TanStack Query는 이 서버 상태를 위한 라이브러리입니다.

---

## 6-3. `taskApi.js`

API 호출을 한 군데에 모았습니다.

이것은 실무적으로 매우 중요합니다.

왜냐하면 컴포넌트가 직접 fetch를 남발하면:

- URL이 흩어지고
- 에러 처리 규칙도 흩어지고
- 교체 비용이 커집니다

API 레이어는 일종의 "통신 어댑터"입니다.

---

## 6-4. `useTasksQuery.js`

여기서 `useQuery`를 감각적으로 이해해야 합니다.

`useQuery`는 단순 fetch 함수가 아닙니다.

그 이상입니다.

- queryKey로 캐시 식별
- queryFn으로 데이터 획득
- 로딩 상태 제공
- 에러 상태 제공
- 캐시 저장
- refetch 관리

즉,
`useQuery`는 **fetch + cache + sync state machine**에 가깝습니다.

---

### queryKey는 왜 중요한가

`["tasks", filters]`

이 key는 "이 서버 데이터의 정체성"입니다.

다르게 말하면:

- 어떤 결과를 캐시할지
- 어떤 데이터를 무효화할지
- 어떤 데이터가 같은 요청인지

를 구분하는 주소입니다.

---

## 6-5. `useTaskMutations.js`

여기서는
- 생성
- 수정
- 토글
- 삭제

를 각각 mutation으로 관리합니다.

### 왜 useMutation인가

조회(read)는 `useQuery`
쓰기(write)는 `useMutation`

이렇게 구분하면 책임이 명확해집니다.

조회는 캐시가 중요하고,
쓰기는 성공/실패/재시도/후속 invalidate가 중요합니다.

---

### optimistic update를 꼭 이해하자

삭제나 토글 시 서버 응답을 기다리기 전에
화면을 먼저 바꾸는 전략이 optimistic update입니다.

장점:
- 사용자 반응성이 좋아짐

단점:
- 서버 실패 시 롤백 필요

따라서 이 패턴은
**빠름과 정확성 사이의 trade-off를 조절하는 기술**입니다.

---

## 6-6. `useRenderCount.js`

이 hook은 학습용으로 매우 중요합니다.

`useRef`는 렌더 사이에서 값이 유지되지만,  
값이 바뀌어도 재렌더를 일으키지 않습니다.

이것은 `useState`와 다릅니다.

### useState
- 값 변경 시 재렌더

### useRef
- 값 유지 가능
- 값 변경 시 재렌더 없음

즉, ref는
"렌더를 다시 일으킬 필요 없는 기억장소"입니다.

# 7. React Hook를 코드 수준에서 이해하기

---

## 7-1. useState

`useState`는 컴포넌트 함수가 렌더될 때마다
새 변수처럼 사라지지 않도록 React가 내부 저장소에 보관하는 state입니다.

중요한 점:

- state 값은 함수 지역변수처럼 보여도,
  실제로는 React 내부 구조에 저장됩니다.
- setter를 부르면 React가 다시 렌더링을 예약합니다.

즉, `useState`는 단순 JS 변수 저장이 아니라  
**렌더 사이를 잇는 프레임워크 관리 상태 슬롯**입니다.

---

## 7-2. useReducer

`TaskForm`에서는 reducer를 사용했습니다.

이유:
- 폼 상태가 객체 형태
- 여러 필드가 함께 움직임
- 상태 전이 규칙을 분리하고 싶음

`reducer(state, action)`는
사실상 작은 상태 머신(state machine)입니다.

```js
state + action => nextState
```

이 방식은
규모가 조금 커지는 순간 `useState` 여러 개보다 정리력이 좋아집니다.

---

## 7-3. useEffect

Effect는 "렌더 이후의 부수효과"입니다.

예:
- DOM focus
- 이벤트 리스너 등록
- 네트워크 후속 처리
- 외부 시스템 동기화

중요한 점:
React는 렌더 단계에서 순수 계산을 기대합니다.

즉:
- JSX를 반환하는 과정은 가급적 순수해야 함
- DOM 직접 조작이나 외부 세계와의 상호작용은 effect로 빼야 함

`TaskForm`에서 focus를 effect에 둔 이유가 바로 이것입니다.

---

## 7-4. useRef

`useRef`는 두 가지 용도로 자주 씁니다.

1. DOM 참조
   - input focus
2. 렌더 사이 값 저장
   - render count
   - previous value
   - mutable cache

초보자에게 자주 생기는 오해:
"ref도 state처럼 값이 바뀌면 화면이 바뀌나요?"

아니요.

ref는 React에게 "다시 그려"라고 말하지 않습니다.

즉:
- state = 렌더를 유발하는 값
- ref = 렌더를 유발하지 않는 값

---

## 7-5. useMemo

`useMemo`는 값을 메모이제이션합니다.

언제 쓸까?

- 계산 비용이 있을 때
- 참조 안정성이 중요할 때
- 하위 memo 컴포넌트와 함께 사용할 때

이번 프로젝트에서는 두 가지 용도로 사용했습니다.

1. `filters` 객체 참조 안정화
2. `stats` 계산 재사용

중요:
useMemo는 무조건 쓰는 최적화가 아닙니다.
과용하면 오히려 복잡도만 늘 수 있습니다.

---

## 7-6. useCallback

`useCallback`은 함수를 메모이제이션합니다.

왜 필요한가?

JS에서 함수는 객체입니다.
즉, 렌더할 때마다 새로 선언되면
참조가 달라집니다.

이것이 왜 문제냐면,
`React.memo`는 props를 비교할 때 참조도 봅니다.

따라서 함수 props가 매번 새로 생기면
memo된 자식도 다시 렌더될 수 있습니다.

즉:

- `React.memo`로 자식 memo
- 부모에서 `useCallback`으로 함수 참조 안정화

이 둘은 함께 쓰일 때 효과가 큽니다.

---

## 7-7. useDeferredValue

이 hook은 초보자에게 조금 낯설 수 있지만, 굉장히 실무적입니다.

검색창 예를 들어봅시다.

사용자는 타이핑을 즉시 보고 싶습니다.
하지만 그 타이핑마다 무거운 필터링/서버 요청/리렌더가 붙으면
입력 체감이 무거워질 수 있습니다.

`useDeferredValue`는
"덜 급한 갱신"을 조금 늦게 반영하여
UI 반응성을 높이는 도구입니다.

이번 프로젝트에서는
- searchInput: 즉시 바뀌는 입력값
- deferredSearchText: 조금 늦게 반영되는 값

을 분리했습니다.

# 8. React 최적화 핵심 개념

---

## 8-1. React.memo

`TaskRow`에 `React.memo`가 붙어 있습니다.

효과:
- props가 같으면 재렌더 생략 가능

하지만 조건이 있습니다.
props 중 함수나 객체가 매번 새로 만들어지면 효과가 떨어집니다.

그래서 최적화는 보통 세트로 갑니다.

- `React.memo`
- `useCallback`
- `useMemo`

---

## 8-2. 참조 동일성(reference equality)

초보자에게 가장 중요한 최적화 개념입니다.

JS에서 객체/배열/함수는 참조형입니다.

즉, 값이 같아 보여도

```js
{} !== {}
[] !== []
() => {} !== () => {}
```

입니다.

React memo 비교에서 이 참조가 중요합니다.

따라서 부모가 렌더될 때마다 새 객체/새 함수가 생기면  
자식 입장에서는 "props가 바뀐 것처럼" 보일 수 있습니다.

---

## 8-3. 최적화는 "무조건"이 아니다

초보자는 종종 모든 것에 useMemo/useCallback을 붙이고 싶어 합니다.

하지만 최적화에는 비용이 있습니다.

- 코드 복잡도 증가
- 가독성 저하
- 잘못 쓰면 이득이 거의 없음

좋은 기준:

1. 실제로 하위 memo 컴포넌트가 있는가?
2. 계산 비용이 의미 있게 큰가?
3. 리스트가 크거나 렌더 빈도가 높은가?

이번 예제는 "학습용이므로 의도적으로 최적화 포인트를 드러낸 것"입니다.

---

# 9. TanStack Query를 왜 써야 하는가

초보자 CRUD는 보통 이렇게 시작합니다.

```js
const [tasks, setTasks] = useState([]);
```

그리고 fetch 후 `setTasks(data)`를 합니다.

작은 프로젝트에서는 가능하지만, 조금만 커지면 문제가 생깁니다.

- 언제 다시 받아와야 하는가?
- 새로고침 시 stale 여부는?
- 여러 화면에서 같은 데이터 공유는?
- mutation 후 최신화는?
- loading / error / cache / retry는?

TanStack Query는 이 문제를 해결합니다.

즉,
로컬 UI 상태는 React state로,
서버 상태는 Query로 나누는 것이 좋습니다.

# 15. 최종 요약

이 프로젝트에서 반드시 가져가야 할 핵심은 다음 10개입니다.

1. Spring Boot에서 객체 생성/연결의 제어권은 IoC 컨테이너가 가진다.
2. 생성자 주입은 가장 명확하고 권장되는 DI 방식이다.
3. Controller, Service, Repository를 분리하면 책임이 선명해진다.
4. Entity와 DTO를 분리해야 경계가 보호된다.
5. React에서 로컬 UI 상태와 서버 상태는 다르게 다뤄야 한다.
6. 서버 상태는 TanStack Query가 매우 강력한 선택지다.
7. useState, useReducer, useEffect, useRef는 역할이 다 다르다.
8. useMemo와 useCallback은 참조 안정성과 비용 최적화에 사용한다.
9. React.memo는 단독보다 useCallback/useMemo와 함께 쓸 때 빛난다.
10. 좋은 풀스택 구조는 "백엔드와 프론트엔드를 각각 그 프레임워크답게" 짜는 것이다.

# 16. 프론트엔드·백엔드 설치와 빌드

이 절에서는 **처음 이 저장소를 받은 사람**이 막히지 않도록, 터미널에서 무엇을 치면 되는지까지 단계별로 정리한다.  
용어부터 짧게 짚고 간다.

- **설치(install)**: 이 프로젝트가 쓰는 **외부 라이브러리**를 내 PC로 내려받아 두는 과정이다. (백엔드는 Maven, 프론트는 npm이 담당한다.)
- **빌드(build)**: 소스 코드를 **배포 가능한 형태**(JAR, 정적 파일 폴더 등)로 묶는 과정이다.
- **실행(run / dev)**: 애플리케이션을 **켜서** 브라우저나 API 클라이언트로 확인하는 과정이다.

이 저장소는 루트 아래에 **`backend/`**(Spring Boot)와 **`frontend/`**(Vite + React)가 나란히 있다. 두 폴더는 **각각 따로** 명령을 실행한다.

## 16.2 백엔드(Spring Boot) — 설치·빌드·실행

### 16.2.1 폴더로 이동

터미널을 연 뒤, **저장소 루트**에서 `backend`로 들어간다.

```bash
cd backend
```

(Windows에서 경로가 다르면, 탐색기로 `backend` 폴더를 연 다음 주소창에 `cmd` 또는 `powershell`을 입력해 그 위치에서 터미널을 열어도 된다.)

### 16.2.2 의존성 설치 겸 빌드 — `mvn clean install`

처음 한 번은 아래를 실행한다.

```bash
mvn clean install
```

의미를 나누면 다음과 같다.

- **`clean`**: 이전 빌드 결과물(`target` 폴더 등)을 비우고 깨끗이 시작한다.
- **`install`**: 컴파일·테스트·패키징까지 하고, 로컬 Maven 저장소에 결과를 반영한다.

인터넷이 연결되어 있어야 하며, 처음에는 **시간이 꽤 걸릴 수 있다**(라이브러리를 많이 받기 때문).

테스트를 건너뛰고 싶다면(환경 문제로 테스트만 실패할 때 등):

```bash
mvn clean install -DskipTests
```

### 16.2.3 개발 중 서버 켜기 — `mvn spring-boot:run`

API와 화면을 같이 보려면 백엔드를 켠 상태가 필요하다.

```bash
mvn spring-boot:run
```

정상 기동이면 로그에 Spring 관련 메시지가 이어지고, **서버 포트는 8080**이다(`application.yml`의 `server.port`).

- **끄는 방법**: 해당 터미널에서 `Ctrl + C`.

### 16.2.4 배포용 JAR만 만들기 — `mvn package`

실행 파일(JAR)만 필요할 때:

```bash
mvn clean package -DskipTests
```

성공하면 `backend/target/` 아래에 `study-backend-0.0.1-SNAPSHOT.jar` 같은 이름의 JAR가 생긴다. 실행 예:

```bash
java -jar target/study-backend-0.0.1-SNAPSHOT.jar
```

(정확한 파일명은 `target` 폴더를 열어 확인한다.)

### 16.2.5 백엔드에서 자주 막히는 것

- **`mvn`을 찾을 수 없음**: Maven이 PATH에 없다. Maven 설치 후 터미널을 다시 연다.
- **Java 버전 오류**: JDK 21이 아닌 버전이 잡혀 있다. `java -version`으로 확인하고, IDE/터미널이 가리키는 JDK를 21로 맞춘다.
- **포트 8080 사용 중**: 다른 프로그램이 8080을 쓰고 있다. 그 프로그램을 끄거나, `application.yml`에서 포트를 바꾼 뒤 프론트 `vite.config.js`의 `proxy.target`도 같은 주소로 맞춘다.

## 16.3 프론트엔드(Vite + React) — 설치·빌드·실행

### 16.3.1 폴더로 이동

새 터미널 탭/창을 열고(백엔드가 돌아가는 창은 그대로 두고), 저장소 루트에서:

```bash
cd frontend
```

### 16.3.2 의존성 설치 — `npm install`

**최초 1회**, 또는 `package.json`이 바뀐 뒤에 실행한다.

```bash
npm install
```

의미: `package.json`에 적힌 패키지(React, Vite, TanStack Query 등)를 **`node_modules` 폴더**로 내려받는다. 용량이 꽤 커질 수 있다.

### 16.3.3 개발 서버 켜기 — `npm run dev`

```bash
npm run dev
```

브라우저에서 **`http://localhost:5173`** 으로 접속한다. 이 프로젝트의 Vite 설정은 **5173 포트**를 쓴다.

- **끄는 방법**: `Ctrl + C`.

### 16.3.4 왜 백엔드를 먼저 켜야 할까?

개발 모드에서 프론트는 `5173`에서 돌아가고, API는 `8080`의 Spring Boot가 처리한다.  
`frontend/vite.config.js`에 **프록시**가 있어서, 브라우저는 `/api/...`로 `5173`에 요청해도 개발 서버가 **`http://localhost:8080`** 으로 넘겨준다.  
따라서 **할 일 목록 API를 쓰려면 백엔드(8080)가 켜져 있어야** 한다. 백엔드를 끈 상태면 네트워크 오류나 빈 화면에 가깝게 보일 수 있다.

### 16.3.5 배포용 정적 파일 빌드 — `npm run build`

```bash
npm run build
```
* 순수 css 파일을 사용했지만, react기반의 jsx파일은 js파일로 변환해야 한다. 
* 그래서 build가 필요하다. 
* package.json 파일안에 build = "vite build" 로 설정되어 있으므로, 실행시 npm run vite build 명령을 실행하게 된다. 

성공하면 `frontend/dist/` 폴더에 배포용 HTML/JS/CSS가 모인다.

로컬에서 빌드 결과만 빠르게 확인하려면:

```bash
npm run preview
```

(빌드 후에 실행하는 것이 일반적이다.)

### 16.3.6 프론트에서 자주 막히는 것

- **`npm`을 찾을 수 없음**: Node.js 미설치 또는 PATH 문제다.
- **`EACCES` / 권한 오류**: 전역 설치가 아니라 이 폴더에서만 `npm install`을 쓰면 대부분 해결된다. 그래도 안 되면 Node 재설치를 검토한다.
- **화면은 뜨는데 데이터가 안 온다**: 백엔드가 8080에서 실행 중인지 확인한다.

## 16.4 하루 개발 루틴으로 정리 (추천 순서)

1. 터미널 A: `cd backend` → `mvn spring-boot:run` (8080)
2. 터미널 B: `cd frontend` → `npm install`(처음만) → `npm run dev` (5173)
3. 브라우저에서 `http://localhost:5173` 접속

이 순서면 프록시를 통해 CRUD 화면이 백엔드와 통신하기 쉽다.
